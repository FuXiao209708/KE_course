# 知识工程智能助教 Demo 系统架构设计文档

## 1. 项目概述

本项目旨在为“知识工程”课程构建一个基于大语言模型（LLM）与知识图谱（KG）的智能助教概念验证系统（Demo/PoC）。系统主要面向前端展示与演示录制，优先保证核心交互链路的跑通与视觉呈现的冲击力。项目采用轻量化的纯 Python 技术栈，通过强 Prompt Engineering 与静态数据模拟，替代复杂的本地大模型微调与动态 Agent 路由。

## 2. 核心功能模块 (Happy Path 演示版)

### 2.1 知识图谱动态答疑模块

* **功能描述：** 学生输入知识工程相关问题，系统基于图谱检索上下文，并以启发式助教口吻回复，同时可视化展示相关知识节点。
* **实现策略：**
  * 建立 3-5 个演示专用的“问题剧本”（如：解释描述逻辑、本体构建等）。
  * 后端基于剧本硬编码或通过简单关键词匹配触发预设的 Cypher 查询语句。
  * 前端侧边栏或对话流中渲染相关知识图谱的局部拓扑图。

### 2.2 习题评判与诊断模块

* **功能描述：** 提供精选的知识工程习题，接收学生主观作答，大模型给出评分、思维误区分析及启发式建议。
* **实现策略：**
  * 以本地静态 JSON 文件（Mock Data）替代动态数据库，存储 3-5 道演示习题及标准答案。
  * 调用 LLM 的 JSON 模式（JSON Mode）强制输出结构化评价（包含：得分、逻辑误区、导学建议）。
  * 前端解析 JSON，利用组件（如进度条、状态框）进行美观渲染。

## 3. 技术栈选型

* **前端交互界面：** `Streamlit` (快速构建基于 Python 的现代对话式数据应用)
* **后端核心逻辑：** `Python 3.10+`
* **图谱数据存储与可视化：**
  * 数据存储：`JSON` 文件 + 内存字典（轻量级，无数据库依赖）
  * 图谱可视化：`pyvis` 或 `streamlit-agraph` (在 Streamlit 中嵌入局部图谱)
* **大模型底座：** 商业大模型 API（如 `DeepSeek-V3`、`Qwen-Max` 或 `GPT-4o`），利用官方 SDK（如 `openai-python`）调用。
* **数据存储 (Demo 级)：** `.json` 静态文件统一存储知识图谱、习题库与演示配置。

│   └── 2_习题演练诊断.py # 习题模块前端页面与交互逻辑
│
├── core/                   # 核心后台逻辑模块
│   ├── __init__.py

## 5. 核心模块交互数据流

### 5.1 答疑模块数据流

1. **用户输入：** 用户在 `1_💬_启发式答疑.py` 聊天框输入问题（按演示剧本）。
2. **触发查询：** 页面将输入传递给 `core/data_manager.py`，根据关键词在本地 JSON 数据中查询相关三元组。
3. **图谱渲染：** 利用 `pyvis` 针对抽取的子图生成 HTML，嵌入 Streamlit 界面。
4. **组装 Prompt：** 将"用户问题 + 图谱三元组文本"套入 `core/prompt_templates.py` 中的启发式助教模板。
5. **LLM 响应：** `core/llm_client.py` 发送请求，获取文本回复并在聊天界面渲染。

### 5.2 习题诊断数据流

1. **加载题目：** 页面加载 `data/mock_exercises.json`，用户通过下拉框选择待答习题。
2. **用户作答：** 用户在文本域输入解答。
3. **请求评判：** 页面收集“题目 + 标准答案 + 用户答案”，套入判题 Prompt 模板。
4. **结构化输出：** LLM 返回符合预设 Schema 的 JSON 字符串。
5. **前端解析：** 将 JSON 解析为具体字段，利用 Streamlit 的 `st.progress`（得分）、`st.warning`（思维误区）、`st.info`（引导建议）进行可视化呈现。

## 6. 开发约束与 Demo 纪律

1. **防御性编程降级：** 作为 Demo 系统，不要投入精力处理边缘网络错误或奇葩用户输入。假定演示者严格按照剧本操作，保障“Happy Path”绝对流畅即可。
2. **解耦 Prompt：** 所有的 Prompt 模板必须统一存放在 `core/prompt_templates.py` 中，方便在演示前根据 LLM 实际输出效果进行快速调优，不要散落在页面代码中。
3. **状态管理：** 合理使用 Streamlit 的 `st.session_state` 来保存对话历史和当前题目状态，确保页面重绘时数据不丢失。
4. **视觉优先：** 知识图谱的渲染是本 Demo 最重要的视觉"抓手"，确保 JSON 加载的节点颜色、大小等属性在可视化组件中对比鲜明，体现"知识结构化"特征。
5. **数据驱动调优：** 演示问题、图谱数据、习题均存储在 JSON 文件中，演示前可快速修改和热更新，无需重启后端服务。

## 7. 项目分阶段构建计划

### 7.1 第一阶段（基础框架 ~ 2-3 天）

**目标：** 快速搭建可运行的原型，建立完整的技术栈验证。

| 任务 | 工作量 | 验收标准 |
|------|--------|---------|
| **环境初始化** | 2h | 项目目录结构搭建，`requirements.txt` 确定，`.env` 模板就位 |
| **Streamlit 主框架** | 4h | `app.py` 和两个页面骨架完成，侧边栏导航可跳转 |
| **LLM 客户端** | 4h | `core/llm_client.py` 实现 API 调用逻辑，支持 Mock 模式（用于离线测试） |
| **数据管理模块** | 4h | `core/data_manager.py` 实现 JSON 加载与基础查询函数 |
| **Prompt 模板集合** | 3h | `core/prompt_templates.py` 编写答疑与判题的 System Prompt 初稿 |
| **数据文件创建** | 3h | 创建 3 个空 JSON 文件（`knowledge_graph.json`、`mock_exercises.json`、`demo_config.json`），填充演示数据结构 |
| **集成测试** | 2h | 手工走通一遍两个页面的 Happy Path（假设 LLM Mock 返回预设数据） |

**阶段产物：** 可运行的原型应用，支持离线测试，LLM 调用代码已就位但使用 Mock。

---

**目标：** 实现两个核心功能模块的完整交互链路。

| 任务 | 工作量 | 验收标准 |

### 7.4 关键卡点与应急预案

| 卡点 | 风险 | 应急方案 |
|------|------|---------|
| **LLM API 调用失败/超时** | 演示中断 | 预加载 3-5 个高质量的 Mock 回复，API 失败时自动降级到 Mock |
| **Prompt 输出质量不达预期** | 演示效果差 | 准备备用 Prompt 版本，演示前多轮调试，必要时用 Mock 数据补救 |
| **pyvis 图渲染卡顿** | 视觉感受差 | 限制图规模（子图最多 15 个节点），优化渲染参数 |
| **环境部署问题（Python 版本、包依赖）** | 无法启动 | 提供 `requirements-lock.txt`（固定版本号），准备虚拟环境一键部署脚本 |
| **演示前数据临时调整** | 时间紧张 | JSON 文件实时可编辑，Streamlit 支持热重载，无需重启 |
